discord_loader.cpp · integration layer

1. Зачем отдельный discord_loader.cpp

Discord SDK состоит из C++ wrapper-кода и нативной discord_game_sdk.dll. Если обычный import DLL не разрешится при загрузке plugin, Windows может отклонить всю DragonbornPresence.dll до того, как наш лог объяснит причину. Проект использует delay-load и собственную проверку.

Loader выполняет только системную работу: не знает о PlayerSnapshot, Activity и правилах STB. Его публичный namespace — DragonbornPresence::detail.

2. Ошибки и сообщения loader

SetFailureReason, строки 17–24

if (!target) return;
try { target->assign(reason); } catch (...) {}

Output pointer необязателен. Assignment string может выделить память и бросить; функция noexcept глотает ошибку. Сам bool результата остаётся главным каналом отказа.

WindowsErrorDetails и DescribeWindowsError

Два string_view смотрят на static literals. constexpr switch декодирует: FILE/PATH_NOT_FOUND, ACCESS_DENIED, INVALID_HANDLE, OUTOFMEMORY, INVALID_DATA, MOD/PROC_NOT_FOUND, BAD_EXE_FORMAT, DLL_INIT_FAILED и unknown.

Некоторые Win32 codes объединены двумя case labels, например memory errors имеют одно объяснение.

SetWindowsFailureReason, строки 61–79

Добавляет operation, числовой DWORD, стабильное имя и объяснение. std::format может бросить, поэтому catch откатывается к короткому operation через безопасный helper.

3. Разбор command и путь рядом с plugin

ExtractExecutablePath(wchar_t*)

  1. Пропускает leading whitespace через std::iswspace.
  2. Пустой command → null.
  3. Если первый символ quote, ищет closing quote через wcschr.
  4. Заменяет closing quote на null terminator и возвращает начало path.
  5. Без quotes идёт до первого whitespace и также вставляет terminator.

Функция принимает mutable pointer, потому что режет registry buffer на месте. Новой string и allocation не требуется.

IsExistingFile

GetFileAttributesW должен вернуть не INVALID_FILE_ATTRIBUTES, а bit FILE_ATTRIBUTE_DIRECTORY должен отсутствовать. Существующий каталог не подходит вместо executable/DLL.

4. IsDiscordRunning(), строки 128–245

Registry: два прохода

Key HKEY_CLASSES_ROOT\discord\shell\open\command хранит команду protocol handler. Первый RegGetValueW вызывается без buffer и возвращает byte count. После проверки создаётся wstring, второй вызов читает value и сообщает type.

Flags допускают REG_SZ и REG_EXPAND_SZ. Код получает valueType, но отдельно его дальше не использует: допустимые типы уже ограничены flags API.

Проверка executable

Command режется helper-функцией. Путь обязан указывать на existing file. Затем wcsrchr оставляет basename процесса; если separator нет, используется весь path.

Snapshot процессов

const HANDLE snapshot = CreateToolhelp32Snapshot(TH32CS_SNAPPROCESS, 0);
PROCESSENTRY32W process{};
process.dwSize = sizeof(process);
if (!Process32FirstW(snapshot, &process)) { /* close + error */ }
do {
    if (_wcsicmp(process.szExeFile, executableName) == 0) {
        found = true;
        break;
    }
} while (Process32NextW(snapshot, &process));
CloseHandle(snapshot);
  • HANDLE проверяется против специального INVALID_HANDLE_VALUE, не только null.
  • Windows требует заполнить dwSize для versioning структуры.
  • _wcsicmp сравнивает wide strings без учёта регистра.
  • Handle закрывается и в error branch, и после цикла.

Exception boundary

Весь метод обёрнут try/catch, потому что wstring и format могут выделять память. Вложенные try защищают построение подробной причины. Результат любого исключения — false.

Что метод не делает: не запускает Discord, не скачивает его, не исправляет registry и не принимает любой процесс с похожим именем.

5. LoadDiscordSdk(), строки 247–278

  1. GetModuleHandleW возвращает true, если SDK уже загружена.
  2. Строится sibling path рядом с plugin.
  3. Файл обязан существовать.
  4. LoadLibraryExW вызывается с двумя search flags.
  5. При ошибке GetLastError() декодируется.

LOAD_LIBRARY_SEARCH_DLL_LOAD_DIR разрешает dependencies из каталога загружаемой DLL. LOAD_LIBRARY_SEARCH_DEFAULT_DIRS использует безопасные default directories вместо legacy current-directory search.

6. DiscordDelayHook, строки 282–301

MSVC delay helper вызывает hook на нескольких стадиях. Код отвечает только когда:

  • dliNotify == dliNotePreLoadLibrary;
  • структуры и имя DLL не null;
  • имя равно discord_game_sdk.dll без учёта регистра.

Return null означает «hook не обработал, продолжить стандартный механизм». Для Discord возвращается загруженный HMODULE, приведённый к ожидаемому FARPROC.

extern "C" const PfnDliHook __pfnDliNotifyHook2 = DiscordDelayHook;

Имя переменной зарезервировано MSVC delayimp. C linkage обеспечивает точное linker name. const запрещает сменить function pointer runtime.

9. RunCallbacks()

  1. Неактивный client сразу false.
  2. Discord process проверяется на каждом tick.
  3. core_->RunCallbacks() обёрнут двумя catch.
  4. Result не Ok → DisableForResult.
  5. Если Activity callback поставил transportHealthy_=false, client отключается.
  6. Без pending signature timeout counter сбрасывается.
  7. С pending counter растёт; 20 ticks отключают зависший callback.

Callback UpdateActivity исполняется внутри RunCallbacks, поэтому после вызова проверяется health, изменённый асинхронной lambda.

IsActive() const noexcept

return core_ && transportHealthy_;

unique_ptr имеет explicit bool conversion. Логическое AND возвращает обычный bool.

10. UpdateActivity(), строки 944–1034

Admission control

Неактивный client или непустая pending signature → false. Одновременно разрешена ровно одна операция.

Локальные owning strings

Четыре human-readable поля проходят LimitForDiscord и хранятся до конца метода. Asset keys не ограничиваются этим helper и берутся как views payload.

Сигнатура без неоднозначности

std::size_t signatureSize = activityFields.size();
for (const auto field : activityFields) signatureSize += field.size();
std::string signature;
signature.reserve(signatureSize);
for (const auto field : activityFields) {
    signature.append(field);
    signature.push_back('\0');
}

Начальные шесть bytes capacity учитывают separators. Null bytes допустимы внутри std::string, потому что string хранит length отдельно. Они различают наборы полей с одинаковой простой concatenation.

Duplicate suppression

Сравнение идёт с последней подтверждённой signature. Pending блокируется раньше. Если Activity не изменился, false означает «ничего не queued», а не ошибку.

Построение SDK Activity

  • discord::Activity activity{} value-initializes wrapper.
  • Type Playing.
  • Timestamp start получает стабильное member field.
  • Пустые text/assets не устанавливаются.
  • c_str() используется для owning strings; data() для asset views.

Асинхронная lambda

[this, signature = std::move(signature)](discord::Result result) noexcept
  • this даёт доступ к members.
  • Init-capture перемещает signature внутрь closure; она живёт до callback.
  • Параметр result приходит от SDK.
  • noexcept и внутренние catch не выпускают исключения в SDK.

В начале callback pending state очищается. При Ok captured signature копируется в last (captured lambda operator по умолчанию const, поэтому здесь assignment из const value выполняет копию), пишется success. При ошибке health false и decoded log.

Lifetime this: callback захватывает pointer client. Текущий client является полем глобального coordinator и живёт весь срок DLL; scheduler прекращает работу до разрушения. Нельзя перенести client во временный local object.

11. Логи, отключение и поля client

Shutdown

Тонкая публичная обёртка над private Disable. Coordinator не получает доступ к деталям transport state.

LogResultFailure

Получает decoder details, пишет operation, name, numeric code, explanation и обещание, что Skyrim продолжит. Catch-all защищает error path от вторичной ошибки logger.

LogFailure

То же для текстовых/WinAPI/C++ причин, у которых нет discord::Result.

DisableForResult

Сначала health false, затем pending очищается и counter=0, ошибка логируется, core reset. Порядок не допускает следующего вызова SDK после зафиксированного отказа.

Disable

hadWork предотвращает повторный одинаковый shutdown log для уже пустого client. Затем состояние обнуляется и core_.reset() запускает destructor SDK core.

CurrentUnixTimestamp

system_clock::now().time_since_epoch() даёт duration; duration_cast<seconds> отбрасывает доли; count() возвращает целое Discord Timestamp.
