adapters/discord

Безопасная загрузка и публикация Discord Presence

Application передаёт нейтральный PresencePayload. Discord adapter проверяет desktop client, загружает bundled SDK, строит Activity и окончательно отключается при неизвестном состоянии транспорта.

1. Application port

DiscordPresenceClient реализует операции:

Initialize(Config)
RunCallbacks()
IsActive()
UpdateActivity(PresencePayload)
Shutdown(reason)

В интерфейсе нет discord::Core, discord::Activity, Windows handles или DLL loading. Благодаря этому coordinator tests используют простой fake transport.

2. Проверка Discord Desktop

IsDiscordRunning не ищет процесс только по фиксированному имени. Он:

  1. читает обработчик протокола discord:// из Windows registry;
  2. извлекает путь executable из command line;
  3. проверяет существование файла;
  4. сравнивает зарегистрированное имя с process snapshot.

Плагин не устанавливает, не восстанавливает и не запускает Discord. Отсутствующий или остановленный клиент выключает интеграцию до обращения к SDK.

3. Bundled DLL и delay-load

DiscordSdkLoader.cpp строит путь к discord_game_sdk.dll рядом с самим SKSE-плагином, проверяет файл и вызывает LoadLibraryExW. Delay-load failure hook возвращает только уже загруженный module handle.

Это предотвращает ранний отказ загрузки всей DragonbornPresence.dll, когда runtime DLL отсутствует. Ошибка превращается в понятный log, а Skyrim продолжает работу.

4. Инициализация SDK

DiscordPresenceClient::Initialize выполняет полный gate:

  1. проверяет Config::enabled;
  2. повторно проверяет Discord Desktop непосредственно перед SDK creation;
  3. загружает bundled DLL;
  4. создаёт discord::Core с DiscordCreateFlags_NoRequireDiscord;
  5. устанавливает SDK log hook;
  6. сохраняет один Unix timestamp текущей игровой сессии;
  7. помечает transport active только после полного успеха.

Повторная process check нужна из-за промежутка между SKSEPluginLoad и kDataLoaded.

5. Построение Activity

UpdateActivity копирует и ограничивает четыре отображаемые строки до 127 UTF-8 bytes, затем назначает:

PayloadDiscord Activity
detailsactivity.SetDetails
stateactivity.SetState
largeImage/largeTextLarge asset и hover text
smallImage/smallTextSmall asset и hover text
session timestampActivity start time

Signature включает все шесть payload fields. Она подавляет payload, совпадающий с последним успешным или уже ожидающим update. Любое новое SDK-поле также должно войти в signature.

6. Один update в полёте

После ActivityManager().UpdateActivity клиент хранит pendingActivitySignature_. Пока callback не завершён, следующий update не отправляется. RunCallbacks обслуживает SDK и отслеживает timeout.

no pending callback → submit → pending signature
pending callback    → RunCallbacks only
success callback    → pending becomes last successful
error/timeout       → Disable()

Coordinator продолжает polling, поэтому актуальное состояние будет повторно предложено после завершения предыдущего callback.

7. Fail closed

После SDK result error, callback error, timeout или exception client:

  • помечает transport unhealthy;
  • очищает pending state;
  • освобождает discord::Core;
  • пишет operation, имя result, числовой код и объяснение;
  • не пытается reconnect в текущем процессе.

Это сознательная политика: продолжать вызывать SDK после неизвестного сбоя опаснее, чем отключить необязательный Presence.
