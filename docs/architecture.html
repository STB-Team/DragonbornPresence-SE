Чистая архитектура вокруг Skyrim и Discord

Как устроен текущий код плагина

Версия 3.1.5 разделяет чистые модели и правила, application orchestration и внешние адаптеры. Skyrim True Believer — единственный игровой runtime; универсального Skyrim adapter в проекте нет.

C++23 Visual Studio 2026 SKSE + CommonLibSSE-NG Discord Game SDK 3.2.1 Catch2 + GitHub Actions

1. Назначение архитектуры

DragonbornPresence загружается внутрь процесса Skyrim через SKSE. Он читает формы и состояние игрока, формирует стабильный Presence и передаёт его Discord Game SDK. Ошибка плагина не должна завершать Skyrim, а код application-слоя не должен зависеть от типов RE::*, SKSE::* или discord::*.

Граница построена по правилу зависимостей: внутренние слои определяют модели и интерфейсы, внешние адаптеры реализуют эти интерфейсы для конкретной инфраструктуры.

Область поддержки: игровой адаптер называется SkyrimTrueBeliever, потому что формы, Papyrus scripts, камни и fallback-тексты относятся к текущей сборке STB.

  • core содержит только принадлежащие приложению значения: Config, PlayerSnapshot, LocationContext, PresencePayload и RefreshReason.
  • application зависит от core и портов. Он не включает RE/SKSE, WinAPI или Discord headers.
  • Адаптеры зависят от application-контрактов и переводят внешние типы в обычные строки, числа, флаги и core-модели.
  • Composition root — единственное место, где выбираются конкретные реализации портов.
Запрещённая зависимость: добавление RE::PlayerCharacter*, discord::Activity или SKSE::TaskInterface в core либо application разрушает границу и делает чистые тесты зависимыми от игры.

4. Composition root и владение

src/DragonbornPresence.cpp создаёт конкретные объекты в порядке их зависимостей:

JsonConfigProvider
StbGameDataSource
DiscordPresenceClient
SkseLogger
PresenceCoordinator(config, gameData, presence, logger)
StbRuntimeAdapter(coordinator, logger)

Все зависимости координатора — non-owning references. Глобальные адаптеры живут дольше него. StbRuntimeAdapter объявлен последним и уничтожается первым: его std::jthread останавливается и присоединяется до уничтожения coordinator и transport.

Публичный заголовок DragonbornPresence.h экспортирует только LoadConfig(), RegisterGameEventHandlers() и OnGameLoaded(). Каждая функция noexcept и не выпускает исключения обратно в SKSE.

5. Жизненный цикл

  1. SKSEPluginLoad. Проверяется входной указатель, вызывается SKSE::Init, создаётся file logger.
  2. Проверка Discord. IsDiscordRunning проверяет protocol handler, executable и процесс. Плагин не скачивает и не запускает Discord.
  3. Конфигурация. LoadConfig читает JSON либо оставляет безопасные defaults.
  4. Messaging listener. main.cpp подписывается на сообщения SKSE.
  5. kDataLoaded. Runtime adapter проверяет UI, event source и TaskInterface; coordinator инициализирует Discord transport, кэширует формы STB и отправляет loading Presence.
  6. kNewGame / kPostLoadGame. Игра помечается готовой, читается первый полный snapshot и публикуется Presence.
  7. Runtime. Каждые 500 мс scheduler ставит не более одной задачи в главный поток Skyrim. Она обрабатывает Discord callbacks и обновляет Presence.
  8. Ошибка или shutdown процесса. Новые операции прекращаются; transport освобождается только в безопасном порядке владения.

6. Application coordinator

PresenceCoordinator получает четыре порта:

ПортЧто coordinator может делатьЧего он не знает
IConfigProviderПолучить полный core::Config.Файл, JSON и путь SKSE.
IGameDataSourceИнициализировать данные и получить PlayerSnapshot.TESForm, Papyrus и STB.esp.
IPresenceClientИнициализировать transport, обработать callbacks, отправить payload, остановить transport.Discord core, activity и DLL.
ILoggerЗаписать info, warning, error или critical message.spdlog и SKSE backend.

Координатор хранит состояния active, permanentlyStopped, gameLoaded, loading, последнее состояние боя и coalesced combat flag. При playable state он строит строки сложности и lvl-<level> ..., применяет LocationAssetResolver и передаёт готовый PresencePayload в порт.

LocationAssetResolver проверяет правила сверху вниз и возвращает первое полное совпадение; если совпадений нет, используются largeImage и largeText из конфигурации.

7. StbRuntimeAdapter

Runtime adapter владеет MenuEventSink, CombatEventSink, указателем на SKSE::TaskInterface и scheduler thread. Event sinks переводят Skyrim events в простые application-вызовы:

  • Main Menu открыт → OnMainMenuOpened();
  • Loading Menu открыт или закрыт → OnLoadingChanged(bool);
  • combat event → RequestCombatRefresh(involvesPlayer, combatEnded).

RE-объекты не пересекают application boundary. Каждый ProcessEvent имеет собственный try/catch и всегда возвращает kContinue, чтобы ошибка Presence не ломала Skyrim event chain.

8. StbGameDataSource и STB helpers

Initialize() после kDataLoaded кэширует deaths global, difficulty quest и 19 description spells камней. Отсутствующая форма остаётся nullptr и приводит к fallback, а не к аварийному завершению.

ReadPlayerSnapshot() копирует из Skyrim только нужные значения: уровень, смерти, сложность, выбранный камень, иерархию локации, состояние боя и текущую combat target. Core и application не хранят указатели на объекты игры.

  • PapyrusValueReader.h ограничивает допустимые form/effect/alias types через concepts и возвращает std::optional.
  • StbStringEncoding сохраняет корректный UTF-8 и преобразует legacy CP1251 через WinAPI.
  • ParseStoneName читает JSON description spell и использует стабильное STB fallback-имя при ошибке.
Контракт STB: Editor IDs, Form ID quest, plugin name, script/property names и список камней синхронизированы с текущим STB.esp. Их нельзя обобщать или переименовывать без проверки сборки STB.

9. Discord adapter

DiscordSdkLoader

Loader проверяет Discord Desktop до первого SDK-вызова и загружает только bundled discord_game_sdk.dll из каталога SKSE/Plugins. Delay-load hook возвращает уже проверенный module handle; системные ошибки превращаются в диагностический текст.

DiscordPresenceClient

Клиент реализует IPresenceClient, владеет discord::Core через unique_ptr, хранит один session timestamp, signature последнего успешного payload и signature ожидающего callback payload.

  • Одинаковый payload не отправляется повторно.
  • Пока предыдущий update callback не завершён, второй update не ставится.
  • Callback timeout, Discord result error или исчезновение клиента permanently disables transport.
  • Все известные discord::Result декодируются в имя, числовой код и понятное объяснение.

10. Потоки и bounded scheduling

scheduler thread                         Skyrim main thread
----------------                         ------------------
sleep 500 ms
atomic pending false?
  yes → AddTask ────────────────────────→ clear pending
  no  → increment wait counter            PresenceCoordinator::Tick
         warning at 500 ms and every 5 s   RunCallbacks
                                           ReadPlayerSnapshot
                                           UpdateActivity

Scheduler не читает forms и не вызывает Discord core. Он только ставит задачу. Atomic flag гарантирует не более одной pending task: если Skyrim main thread задержан, новые тики объединяются, а память остаётся ограниченной.

На первом пропущенном тике записывается warning; затем он повторяется каждые 10 тиков. Combat events также coalesce в atomic flag и обрабатываются ближайшим общим tick.

11. Ошибки и защитные границы

ГраницаПоведение
SKSEPluginLoadЛовит все исключения, пишет critical/debug output и возвращает false.
Публичный facadeПереводит исключение в HandleException; функции остаются noexcept.
Menu/combat sinksЛовят исключение до возврата в Skyrim и останавливают интеграцию.
Main-thread taskЛовит исключение, останавливает scheduler и coordinator.
Scheduler threadОстанавливает producer, но не уничтожает Discord core на фоновом потоке.
Discord callback/resultTransport становится inactive; повторных SDK-вызовов нет.

Политика — fail closed для Presence и fail safe для Skyrim: после неизвестного состояния transport интеграция не пытается автоматически восстановиться внутри текущего процесса игры.

13. Сборка, deploy и релиз

cmake --preset vs2026
cmake --build --preset release

Production target линкует CommonLibSSE-NG, Discord Game SDK, spdlog, nlohmann/json и delayimp. Release создаёт оптимизированную DLL с PDB, копирует runtime-файлы в заданный preset-каталог MO2 и упаковывает build/vs2026/DragonbornPresence.zip.

.github/workflows/release.yml запускается для тегов v*, проверяет соответствие tag и CMake version, собирает Release ZIP и публикует GitHub Release. Оба workflow используют actions/checkout@v5.

Разделение проверок: CI быстро доказывает engine-independent контракты. Release build доказывает совместимость всех headers, symbols и link dependencies. Запуск через SKSE доказывает runtime integration.

14. Инварианты, которые нельзя ломать

  • Никаких RE/SKSE/Discord типов во внутренних слоях. Только core-модели и application-порты.
  • Игровые данные и Discord core обрабатываются в Skyrim main thread. Scheduler только планирует.
  • Не более одной pending main-thread task. Задержка не создаёт неограниченную очередь.
  • Исключения не пересекают callback или DLL boundary. Каждая внешняя точка имеет catch guard.
  • Runtime adapter уничтожается первым. Его thread не переживает coordinator и adapters.
  • Первое правило локации побеждает. Порядок 438 JSON rules является поведением.
  • Session timestamp стабилен весь запуск. Presence update его не пересоздаёт.
  • Нет скрытого восстановления Discord. Плагин не скачивает, не устанавливает и не запускает клиент или SDK.
  • Игровой namespace — SkyrimTrueBeliever. Второй универсальный skyrim adapter не создаётся.
