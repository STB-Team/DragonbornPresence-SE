1. Что объясняет этот раздел

DragonbornPresence — нативная DLL, которую SKSE загружает внутрь процесса Skyrim. Плагин читает состояние игрока и отправляет его в Discord Rich Presence. Он не является отдельной программой: код работает в адресном пространстве игры, поэтому необработанное исключение, неверный указатель или обращение к игровым объектам из неправильного потока может повредить стабильности Skyrim.

Главная задача архитектуры — разделить четыре разные ответственности:

  1. получить сигнал от SKSE о стадии загрузки игры;
  2. безопасно прочитать формы STB и текущее состояние игрока;
  3. преобразовать данные в ограниченный контракт Discord Activity;
  4. при любой ошибке отключить только интеграцию, не выпуская исключение в Skyrim.
Главный принцип кода: Discord Presence вторичен, Skyrim первичен. Если Discord, конфигурация или внутренний код интеграции отказали, плагин прекращает собственную работу, а игра должна продолжить работу.
Границы документа: здесь разобран весь собственный код репозитория. Внутренности Skyrim, CommonLibSSE-NG и Discord Game SDK объясняются в той мере, в которой плагин ими пользуется; исходники самих внешних библиотек не дублируются.

Сопоставляет позицию игрока с первым подходящим правилом изображения.

Кэширует формы STB и строит снимок игрока.

Владеет Discord core, отправляет Activity и декодирует ошибки SDK.

application

Состояние загрузки, event sinks, scheduler и координация слоёв.

Изолирует WinAPI-проверки и фактическую загрузку DLL.

Публичная граница

LoadConfig, RegisterGameEventHandlers, OnGameLoaded; каждая ловит исключения.

Почему большая реализация находится в одном .cpp

Внутренние типы помещены в анонимный namespace. Они имеют internal linkage и недоступны другим translation unit. Это скрывает детали и оставляет внешнему коду только три функции из DragonbornPresence.h. Цена решения — большой файл; навигацию поддерживают именованные namespaces и классы.

5. Полный жизненный цикл

  1. Windows/SKSE загружает DLL. Вызывается SKSEPluginLoad(skse).
  2. SKSE инициализируется. SKSE::Init(skse) подготавливает интерфейсы CommonLib.
  3. Создаётся лог. InitializeLogging() открывает DragonbornPresence.log в режиме append.
  4. Discord проверяется заранее. Код проверяет регистрацию discord://, путь к EXE и наличие соответствующего процесса.
  5. Читается JSON. LoadConfig() сохраняет валидированную конфигурацию в глобальном координаторе.
  6. Регистрируется Messaging listener. Теперь SKSE может сообщать о стадиях загрузки.
  7. kDataLoaded. Данные игры готовы: проверяются сервисы UI, событий и задач; создаётся Discord core; кэшируются формы STB; отправляется loading activity; регистрируются event sinks; запускается scheduler.
  8. kNewGame или kPostLoadGame. Координатор помечает игру готовой и отправляет полный снимок игрока.
  9. Каждые 500 мс. Scheduler ставит не более одной задачи в game thread. Задача вызывает Discord callbacks и перечитывает Presence.
  10. При изменении. Новый Activity отправляется; неизменный payload отбрасывается по сигнатуре.
  11. При ошибке. Интеграция становится permanently stopped, scheduler получает stop request, Discord core освобождается в безопасной точке.

Последовательность первого рабочего Presence

SKSEPluginLoad
  → LoadConfig
  → SKSE message kDataLoaded
  → DiscordPresenceClient::Initialize
  → StbDataProvider::Initialize
  → SendLoadingPresence
  → SKSE message kPostLoadGame / kNewGame
  → ReadPlayerSnapshot
  → LocationAssetResolver::Resolve
  → DiscordPresenceClient::UpdateActivity
  → следующий RunCallbacks

  → запись "Presence updated"

Состояние координатора

  • active_ — интеграция сейчас может выполнять работу;
  • permanentlyStopped_ — повторная инициализация в этом экземпляре запрещена после фатальной ошибки;

  • gameLoaded_ — существует игровая сессия, данные игрока можно публиковать;
  • lastCombatState_ — помогает заметить завершение боя;
  • callbackTaskPending_ — в очереди уже есть задача scheduler;
  • combatRefreshRequested_ — combat event просит ближайший poll пометить причиной combat.

7. Потоки, очередь задач и события

В проекте есть scheduler thread (std::jthread) и главный поток Skyrim. Scheduler сам не читает PlayerCharacter и не вызывает discord::Core. Он спит 500 мс и через SKSE::TaskInterface::AddTask просит game thread выполнить работу.

scheduler thread                         Skyrim game thread
----------------                         ------------------
sleep_for(500 ms)
callbackTaskPending_.exchange(true)
AddTask(lambda) -----------------------> callbackTaskPending_ = false
                                         Discord RunCallbacks()
                                         ReadPlayerSnapshot()
                                         UpdateActivity()
                                         return
следующий tick

Зачем callbackTaskPending_

Если Skyrim занят загрузкой и не исполняет очередь задач, scheduler продолжает просыпаться. Без защиты он добавлял бы новую lambda каждые 500 мс, очередь и память росли бы. exchange(true) атомарно проверяет и устанавливает флаг: если задача уже ожидает, новая не создаётся.

callbackTaskPendingTicks_ считает пропущенные постановки. Предупреждение пишется примерно через 500 мс ожидания и затем раз в 5 секунд. Это не таймаут game thread и не авария — это наблюдаемая задержка с ограниченной очередью.

Почему std::jthread, а не std::thread

jthread имеет stop_token, поддерживает request_stop() и при уничтожении соединяется с потоком. Код может попросить цикл завершиться без собственного небезопасного флага остановки.

Почему combat event не отправляет Activity сразу

HandleCombatEvent лишь устанавливает atomic-флаг. Следующий общий poll читает фактическое состояние игрока и объединяет несколько событий в один refresh. Это предотвращает очередь из повторяющихся задач и устраняет конкуренцию между event path и периодическим обновлением.

Правило изменения: не переносите чтение Skyrim forms или вызовы Discord core внутрь scheduler lambda до AddTask. Фоновый поток должен только планировать работу.

8. src/main.cpp: вход в плагин

extern "C" IMAGE_DOS_HEADER __ImageBase

__ImageBase — символ текущего PE-модуля. Его адрес приводится к HMODULE, чтобы WinAPI определил путь именно к DragonbornPresence.dll, а не к SkyrimSE.exe. extern "C" отключает C++ name mangling для системного символа.

InitializeLogging()

  1. Просит SKSE::log::log_directory() вернуть стандартный каталог SKSE logs.
  2. Если каталог недоступен, через GetModuleFileNameW берёт каталог DLL; последний fallback — current working directory.
  3. Добавляет имя DragonbornPresence.log.
  4. Создаёт многопоточный basic_file_sink_mt в append-режиме.
  5. Устанавливает уровень info и flush на каждом сообщении уровня info или выше.
  6. Делает logger глобальным для SKSE::log.

Функция noexcept и сама ловит всё. Если файл открыть нельзя, обычного logger ещё нет, поэтому сообщение отправляется в Windows debugger через OutputDebugStringA, а плагин не запускается.

LogEntryPointFailure()

Единый аварийный выход ранней стадии. Пытается написать critical; если даже logging сломан, использует debugger. Функция ничего не бросает обратно в загрузчик SKSE.

OnSkseMessage()

  • kDataLoaded → RegisterGameEventHandlers(). Только теперь формы и игровые сервисы считаются готовыми.
  • kNewGame и kPostLoadGame → OnGameLoaded(). Для новой игры и загруженного сохранения нужен одинаковый первый полный refresh.
  • Другие сообщения игнорируются намеренно.

SKSEPluginLoad()

Это экспортируемая CommonLib-макросом точка входа. Она валидирует LoadInterface*, вызывает SKSE::Init, поднимает лог, проверяет Discord, пишет версию из PluginDeclaration, читает config и регистрирует listener. Возвращаемое true сообщает SKSE, что плагин принят; false прекращает его загрузку.

Важная текущая семантика: если Discord Desktop не запущен уже в SKSEPluginLoad, функция возвращает false. Плагин не остаётся загруженным в ожидании запуска Discord.

9. src/DragonbornPresence.cpp: основной модуль

Файл организован сверху вниз от простых данных к управляющему объекту:

constants → model → text → configuration → assets → game
          → integration → application → публичные noexcept-обёртки

Почти всё находится в анонимном namespace внутри namespace DragonbornPresence. Поэтому внутренние классы не экспортируются из DLL и не создают публичного API. В конце файла существует один глобальный g_presenceCoordinator; он живёт весь срок жизни DLL и владеет всеми подсистемами.

Публичные обёртки

LoadConfig(), RegisterGameEventHandlers() и OnGameLoaded() вызывают одноимённые методы глобального координатора. Каждая функция объявлена noexcept, ловит std::exception и неизвестные исключения и переводит их в HandleException. Это последняя защита на границе с main.cpp и SKSE.

10. Конфигурация: ConfigLoader

Алгоритм Load()

  1. Сразу создаётся model::Config config со всеми defaults.
  2. Открывается Data\SKSE\Plugins\DragonbornPresence.json.
  3. Если файла нет, возвращаются defaults и пишется info.
  4. nlohmann::json::parse(file) разбирает синтаксис JSON.
  5. Корень обязан быть object.
  6. Блоки discord, assets и small_images читаются только как objects.
  7. Каждое корректное поле заменяет свой default; поле неверного типа игнорируется с warning.
  8. Правила локаций сначала собираются во временный vector и только затем целиком перемещаются в config.

Маленькие функции валидации

  • FindObject различает «ключ отсутствует» и «ключ существует, но имеет неверный тип».
  • ReadBool принимает только настоящий JSON boolean, не строки "true" и не числа.
  • ReadString принимает только JSON string.
  • ReadApplicationId принимает положительное целое число или строку только из десятичных цифр. std::from_chars не создаёт исключений и позволяет проверить, что обработана вся строка.
  • ReadLocationImageRules требует array; каждое правило требует непустой image и хотя бы один selector.

Почему временный parsedRules

Объект config не получает частично разобранный vector. reserve(rules->size()) заранее выделяет достаточно места и уменьшает количество reallocations. В конце std::move(parsedRules) передаёт внутренний буфер без копирования каждого правила.

Что сейчас не используется

Поле schema_version присутствует в JSON, но текущий ConfigLoader его не читает и не валидирует. Оно документирует формат для людей, но не влияет на runtime.

Правила локаций

LocationAssetResolver::Resolve идёт по vector сверху вниз. RuleMatches требует совпадения всех непустых селекторов:

  • worldspace — точный Editor ID текущего worldspace;
  • cell — точный Editor ID parent cell;
  • location — точный Editor ID текущей location или одного из её parentLoc;
  • match — подстрока display location с ASCII case folding.

Первое полное совпадение возвращает asset. Пустой text заменяется общим largeText. Если совпадений нет или player отсутствует, возвращается largeImage/largeText.

Почему порядок важен: общее правило, поставленное раньше специального, сделает специальное недостижимым. Самые точные правила должны идти выше общих.

11. Игровые данные: StbDataProvider и ScriptUtils

StbDataProvider::Initialize()

Метод вызывается один раз после kDataLoaded. Он получает TESDataHandler, ищет deaths global по Editor ID, difficulty quest по Form ID и имени plugin, затем ищет 19 description spells камней. Результаты сохраняются в runtimeData_.

Кэш нужен потому, что lookup форм дороже чтения готового указателя и не должен повторяться два раза в секунду. После инициализации лог показывает, какие формы найдены и сколько камней разрешено.

ParseStoneName()

STB хранит описание spell как JSON-текст. Метод вызывает GetDescription, разбирает JSON и берёт строковое поле name. Если spell отсутствует, description пуст, JSON сломан или name непригоден, используется compile-time fallback из kStoneDefinitions.

ReadPlayerSnapshot()

  1. Создаёт snapshot с безопасными начальными значениями.
  2. Получает singleton игрока; при отсутствии возвращает пустой snapshot.
  3. Читает level.
  4. Если deaths global найден, преобразует его float-поле value в int.
  5. Определяет первый активный камень через HasSpell.
  6. Читает сложность из Papyrus alias.
  7. Строит отображаемую позицию.
  8. Проверяет бой и current combat target.

Когда player в бою и target доступен, имя и уровень сохраняются в lastCombatTarget_. Если следующий snapshot всё ещё в бою, но handle временно пуст, Discord продолжает показывать последнюю цель. После выхода из боя cache сбрасывается.

BuildDisplayPosition()

Метод сначала ищет первое непустое имя в цепочке current location → parent location. Затем читает worldspace и cell:

  • если worldspace имеет имя и location отличается — "worldspace: location";
  • если location совпадает с worldspace — только worldspace;
  • без worldspace — location;
  • без location — cell.

ScriptUtils::GetPropertyOrVariable

Это закрытый шаблон для доступа к данным Papyrus:

  1. requires разрешает только TESForm, ActiveEffect или BGSBaseAlias. Ошибочный тип не скомпилируется.
  2. Получается singleton Papyrus Virtual Machine и object handle policy.
  3. Для разных типов определяется правильный VM type ID.
  4. GetHandleForObject связывает C++ объект с Papyrus instance.
  5. BSSpinLockGuard защищает таблицу attachedScripts на время чтения.
  6. Ищется script с точным именем типа.
  7. Сначала ищется property, затем variable.
  8. Возвращается optional<Variable*>, поэтому отсутствие не маскируется нулём.

GetVariableValue<T> с if constexpr выбирает getter во время компиляции: bool, signed/enum, unsigned, float или string. Неподдерживаемый тип вызывает compile-time static_assert, а не runtime-ошибку.

GetFirstAliasScriptPropertyOrVariable ищет первый непустой alias quest и применяет общий reader. В текущем плагине через него читается aaMZ_SelectedLevel_OfDifficulty.

Осторожно: имя Papyrus script сравнивается точно с aamz_mcmdatastorage, а property — с aaMZ_SelectedLevel_OfDifficulty. Изменение имен в STB.esp/Papyrus требует синхронного изменения констант C++.

12. Строки, UTF-8 и CP1251

Зачем нужна конвертация

Discord ожидает UTF-8. Строки Skyrim/STB могут уже быть UTF-8 либо приходить в CP1251. Нельзя безусловно перекодировать всё: повторное преобразование корректного UTF-8 испортит кириллицу.

IsValidUtf8()

Функция проходит по байтам и распознаёт последовательности длиной 1–4 байта. Для continuation bytes проверяется префикс 10xxxxxx. После сборки code point отвергаются:

  • значения выше Unicode 0x10FFFF;
  • UTF-16 surrogate range 0xD800–0xDFFF;
  • overlong encodings, когда символ записан большим числом байт, чем требуется;
  • обрезанные и неправильно начинающиеся последовательности.

Cp1251ToUtf8()

WinAPI выполняет преобразование в два этапа: MultiByteToWideChar(1251) создаёт UTF-16, затем WideCharToMultiByte(CP_UTF8) создаёт UTF-8. Сначала каждый API вызывается без выходного буфера для вычисления размера, затем создаётся строка точного размера. Null terminator исключается из результата.

text::FromGameString()

Null и пустая строка дают {}. Валидный UTF-8 копируется как есть; невалидный считается CP1251 и конвертируется.

text::LimitForDiscord()

Discord-поле ограничивается 127 байтами, не символами. Кириллица и emoji занимают несколько байт. Если byte 127 находится внутри continuation sequence, индекс сдвигается назад к началу code point. Поэтому на выходе не возникает обрезанного UTF-8 символа.

ContainsAsciiInsensitive()

Функция вручную приводит только ASCII A–Z к a–z и ищет pattern во всех offsets. Кириллица не меняет регистр. Следовательно, match с кириллицей должен совпадать по регистру; case-insensitive гарантирован только для ASCII.

13. discord_loader.cpp: Windows и delay-load

Общая цель

Плагин не должен позволить Windows неявно искать Discord DLL по непредсказуемым каталогам и не должен пытаться чинить или запускать Discord. Loader проверяет окружение до первого SDK-вызова и загружает только DLL рядом с плагином.

Маленькие helper-функции

  • SetFailureReason безопасно записывает текст в optional output pointer и глотает allocation exceptions.
  • DescribeWindowsError сопоставляет известные Win32 коды стабильному имени и объяснению.
  • SetWindowsFailureReason формирует единый actionable error с числом, именем и смыслом.
  • ExtractExecutablePath получает первый executable из registry command, учитывая quoted и unquoted путь. Функция изменяет переданный mutable buffer, вставляя null terminator.
  • IsExistingFile проверяет, что путь существует и не является каталогом.
  • BuildPluginSiblingPath берёт путь текущей DLL через __ImageBase и заменяет имя файла на запрошенное.

IsDiscordRunning()

  1. Читает HKEY_CLASSES_ROOT\discord\shell\open\command.
  2. Первый RegGetValueW узнаёт размер registry value.
  3. Второй читает строку в std::wstring.
  4. Из command извлекается путь executable и проверяется файл.
  5. Из пути берётся имя процесса.
  6. CreateToolhelp32Snapshot получает список процессов Windows.
  7. Process32FirstW/Process32NextW ищут имя без учёта регистра.
  8. Каждый HANDLE закрывается через CloseHandle.

Проверка связывает процесс именно с executable, зарегистрированным для discord://, а не просто доверяет фиксированному имени Discord.exe.

LoadDiscordSdk()

Если DLL уже загружена, возвращается true. Иначе строится соседний путь SKSE\Plugins\discord_game_sdk.dll, проверяется файл и вызывается LoadLibraryExW с LOAD_LIBRARY_SEARCH_DLL_LOAD_DIR | LOAD_LIBRARY_SEARCH_DEFAULT_DIRS. Это ограничивает и делает явным DLL search behavior.

DiscordDelayHook

CMake линкует import library с /DELAYLOAD:discord_game_sdk.dll. Поэтому Windows не обязана загрузить SDK при загрузке самого плагина. Переменная __pfnDliNotifyHook2 подключает hook MSVC delay-load helper. На запрос именно Discord DLL hook возвращает уже загруженный module либо загружает соседний файл теми же безопасными flags.

Почему и явный LoadDiscordSdk, и hook: явная функция даёт контролируемую проверку и понятную ошибку до SDK-вызова; hook гарантирует правильный путь в момент разрешения delay-loaded imports.

14. DiscordPresenceClient

Владение и состояния

core_ — unique_ptr, единственный владелец Discord core. transportHealthy_ показывает, можно ли продолжать транспортные вызовы. Клиент активен только когда существуют оба условия.

sessionStartTimestamp_ — const field, инициализированный текущим Unix time при создании глобального координатора. Он не пересоздаётся при каждой activity, поэтому Discord timer показывает длительность текущего запуска.

Initialize(config)

  1. Сбрасывает transport health.
  2. При enabled=false не загружает SDK.
  3. Повторно проверяет Discord и локальную DLL.
  4. Вызывает discord::Core::Create(applicationId, DiscordCreateFlags_NoRequireDiscord, ...).
  5. Временно оборачивает raw pointer в unique_ptr candidate, чтобы не потерять его при ошибке.
  6. Перемещает candidate в core_ только при Result::Ok.
  7. Устанавливает SDK log hook начиная с warning.
  8. Пишет Application ID и стабильный session_start.

RunCallbacks()

Discord Game SDK завершает асинхронные операции только при регулярном вызове callbacks. Перед вызовом код снова проверяет процесс Discord. Любое SDK result кроме Ok, C++ exception или ошибка асинхронного callback окончательно отключает client.

Если есть pendingActivitySignature_, счётчик увеличивается. После 20 тиков × 500 мс = 10 секунд callback считается зависшим, integration отключается. Это ограничивает вечное состояние «запрос в полёте».

UpdateActivity(payload)

  1. Возвращает false, если client неактивен или другой update уже pending.
  2. Ограничивает четыре текстовых поля по UTF-8 bytes.
  3. Собирает сигнатуру из шести полей, разделяя их null-байтом.
  4. Если сигнатура совпадает с последней подтверждённой, Discord не вызывается.
  5. Создаёт discord::Activity, устанавливает type Playing и timestamp.
  6. Заполняет только непустые поля.
  7. Сохраняет pending signature и вызывает ActivityManager().UpdateActivity.
  8. Callback при Ok переносит signature в last и пишет info; при ошибке помечает transport unhealthy.

Null-разделители нужны, чтобы разные наборы полей не дали одинаковую concatenation. Без разделителя ["ab", "c"] и ["a", "bc"] обе превратились бы в "abc".

Почему last обновляется только в callback

Постановка операции ещё не означает, что Discord её принял. lastActivitySignature_ меняется только после Result::Ok; до этого значение находится в pendingActivitySignature_.

Декодирование ошибок

DescribeResult содержит switch по всем известным результатам Discord SDK 3.2.1. Лог получает стабильное enum name, числовой code и человеческое explanation. default защищает от неизвестного значения новой/повреждённой версии SDK.

Disable и DisableForResult

Обе функции сбрасывают health, pending state и core_. После core_.reset() последующих SDK-вызовов не будет. Разница — готовый discord::Result логируется через полный decoder, а текстовая причина через общий LogFailure.

15. PresenceCoordinator и event sinks

Почему coordinator владеет всем

Он гарантирует общий lifecycle: config живёт дольше resolver, data provider живёт между polls, client имеет одного владельца, event sinks содержат ссылку на coordinator, scheduler останавливается вместе с ним.

RegisterGameEventHandlers()

Метод проверяет три обязательных сервиса: RE::UI, ScriptEventSourceHolder и SKSE::TaskInterface. Только после успешной проверки и инициализации Discord выставляет active_=true, кэширует формы, отправляет loading state, подписывает sinks и запускает thread. Такой порядок не оставляет частично зарегистрированный плагин после раннего отказа.

Меню и loading state machine

  • Открытие Main Menu: gameLoaded_=false, loading_=true.
  • Открытие Loading Menu: loading включается.
  • Закрытие Loading Menu после сигнала game loaded: loading выключается и отправляется полный refresh.
  • OnGameLoaded: gameLoaded_=true, loading_=false, немедленный refresh.

RefreshPresence(reason)

Если игра не готова, делегирует loading state. Иначе читает snapshot, форматирует:

details = difficulty
state   = "lvl-<level> 💀-<deaths|—> <stone>"
large   = первое совпавшее location rule или общий fallback
small   = combat asset и target text только во время боя
timer   = неизменный sessionStartTimestamp_

После вызова client coordinator проверяет IsActive(). Если транспорт умер, Stop() выключает дальнейшую работу. Подробная snapshot-строка логируется только если UpdateActivity действительно поставил новый запрос.

MenuEventSink и CombatEventSink

Оба наследуются от RE::BSTEventSink<Event>. Их ProcessEvent проверяет null, вызывает coordinator, ловит все исключения и всегда возвращает kContinue. Это позволяет следующим Skyrim listeners получить событие даже после обработки данным плагином.

Обработка исключений

HandleException используется на внешних/game-thread границах: ставит inactive и permanently stopped, просит thread остановиться, пишет critical и вызывает shutdown client. HandleBackgroundException вызывается из scheduler thread и намеренно не уничтожает Discord core там: он останавливает producer, а ресурс остаётся до безопасного завершения и уничтожения владельца.

16. Заголовки и precompiled header

DragonbornPresence.h

Экспортирует только три noexcept функции. Это минимальный API между main.cpp и основной реализацией. Внутренние классы в header не попадают, поэтому изменение их полей не требует менять пользователей модуля.

AdditionalFunctions.h и discord_loader.h

Содержат объявления функций и минимальные standard includes, необходимые для типов сигнатур. Header не должен рассчитывать, что другой header был подключён раньше.

ScriptUtils.h

Шаблон реализован в header, потому что компилятору нужна полная реализация для создания конкретных версий Get...<int> и других типов в месте использования.

pch.h

Подключает тяжёлый SKSE/Impl/PCH.h один раз в precompiled header и включает литералы вроде "..."sv. PCH уменьшает время компиляции, но не меняет runtime-поведение DLL.

#pragma once

Предотвращает повторную обработку одного header в пределах translation unit. Это защита от повторных объявлений, а не механизм runtime singleton.

17. CMake, зависимости, DLL, PDB и релиз

Основные настройки

  • CMake 4.2, C++23 обязателен.
  • FetchContent фиксирует версии fmt 10.2.1, spdlog 1.13.0, rapidcsv 8.83, nlohmann/json 3.11.3 и CommonLibSSE-NG 3.7.0.
  • Discord Game SDK 3.2.1 скачивается ZIP-файлом в build directory, если header ещё отсутствует.
  • C++ wrapper исходники SDK собираются в статическую библиотеку discord_cpp.
  • add_commonlibsse_plugin создаёт SKSE DLL и PluginDeclaration.

Почему некоторые зависимости выглядят неиспользуемыми

fmt, spdlog и rapidcsv также требуются/обнаруживаются CommonLibSSE. Проект явно подготавливает их до подключения CommonLib, чтобы версии и пути были определены стабильно.

Линковка

Plugin линкуется с discord_cpp, delayimp и nlohmann_json. Опция /DELAYLOAD:discord_game_sdk.dll откладывает загрузку runtime DLL. /utf-8 заставляет MSVC читать исходники с русским текстом и emoji как UTF-8.

PDB в Release

MSVC_DEBUG_INFORMATION_FORMAT ProgramDatabase и /DEBUG:FULL создают полный PDB даже для оптимизированного Release. /OPT:REF удаляет неиспользуемый код, /OPT:ICF объединяет одинаковые функции. PDB нужен для сопоставления адреса crash log с функцией и строкой; он не делает код debug-медленным и сам не предотвращает crash.

Packaging и deploy

После build CMake копирует DLL, PDB, SDK DLL и JSON в staging _package/SKSE/Plugins, создаёт DragonbornPresence.zip, а при заданном DRAGONBORNPRESENCE_DEPLOY_DIR копирует те же runtime-файлы прямо в мод MO2.

CMakePresets.json фиксирует Visual Studio 18 2026, x64, build directory и локальный путь MO2. Presets release и debug собирают target DragonbornPresence.

GitHub Release workflow

Push тега v* запускает Windows runner, выполняет configure и Release build, проверяет ZIP и создаёт GitHub Release через gh release create. Workflow имеет contents: write, потому что без этого token не сможет опубликовать release.

Ключевые сообщения по порядку

Отсутствующая следующая строка локализует этап отказа. Например, есть Registering..., но нет Discord Game SDK initialized — проблема возникла в проверке Discord, загрузке DLL или Core::Create.

Почему snapshot пишется после постановки update

Подробный лог создаётся только когда payload отличается и новый запрос действительно queued. Poll без изменения данных не засоряет файл. Отдельная строка Presence updated появляется позже, когда callback подтвердил успех.

Проверка таймера

Все session_start в одном запуске обязаны совпадать. Изменение числа означает, что timestamp начали создавать на каждый refresh — это регрессия.

Задержка главного потока

Discord task 'RunCallbacks/RefreshPresence' has been waiting on Skyrim's main thread
for about 500 ms; 1 callback request coalesced. Only one task is queued...

Это означает: scheduler жив, но Skyrim ещё не исполнил предыдущую задачу. Число coalesced — сколько дополнительных запросов не было добавлено благодаря bounded queue.

Windows debugger fallback

Если logger не удалось создать, писать в log невозможно. Ранний код использует OutputDebugStringA; такие сообщения видны в Visual Studio debugger или Sysinternals DebugView.

Почему catch есть в нескольких местах

Это не одинаковое дублирование. Каждая внешняя граница должна быть noexcept по поведению: SKSE entry point, SKSE messages, event sinks, queued lambda, scheduler lambda и Discord callback вызываются внешней системой. Исключение нельзя позволить пройти через чужой ABI/callback stack.

Почему после transport error нет retry

Текущий контракт выбирает fail closed: состояние SDK после ошибки может быть неизвестно, повторные вызовы могут зависнуть или повторно ошибаться, а Presence не стоит риска для Skyrim. Для восстановления нужен полный перезапуск игры.

20. Примеры безопасных изменений

Перед изменением: создайте отдельную ветку, соберите Release с PDB, запустите Skyrim через SKSE, воспроизведите изменённый путь и проверьте log. Примеры показывают точки изменения, но названия форм и Discord Asset keys должны существовать в текущем STB/Discord Application.
Пример A: добавить или переупорядочить изображение локации без C++

Измените только config/DragonbornPresence.json и поставьте специальное правило выше общего:

{
  "cell": "MySpecialInteriorCell",
  "image": "my_special_asset",
  "text": "Особое место"
}

Почему этого достаточно: ConfigLoader уже понимает cell, а resolver перебирает правила. После build JSON автоматически копируется в MO2. Полностью перезапустите Skyrim.

Проверка: snapshot log должен содержать ожидаемые location='...' и large='my_special_asset'.

Пример B: изменить формат строки state

Точка изменения — PresenceCoordinator::RefreshPresence:

const std::string stateText = std::format(
    "Ур. {} | Смерти: {} | {}",
    snapshot.level,
    deathsText,
    snapshot.stone);

Не переносите форматирование в DiscordPresenceClient: client должен оставаться универсальным транспортом. LimitForDiscord продолжит применять лимит 127 UTF-8 bytes.

Проверка: загрузить сохранение, увидеть новый state в Discord и одну новую snapshot-запись в log.

Пример C: добавить новый индекс сложности STB
  1. Добавить именованное значение в model::Difficulty.
  2. Добавить case в StbDataProvider::DifficultyName.
  3. Не менять ReadSelectedDifficulty: он уже читает int и передаёт его mapper.
enum class Difficulty : int {
    // существующие значения...
    kNewMode = 5,
};

case model::Difficulty::kNewMode:
    return "Новый режим";

Проверка: реально выбрать значение 5 в STB и убедиться, что details изменился. Не подставляйте 5 только в C++ debugger без проверки источника Papyrus.

Пример D: добавить новый камень

Добавьте строку в compile-time kStoneDefinitions:

StoneDefinition{"MyStoneDescriptionSpell", "Новый камень"},

На kDataLoaded provider найдёт spell, разберёт его description один раз и затем проверит player->HasSpell на каждом snapshot.

Проверка: init log должен увеличить denominator и показать найденный spell; после получения камня state должен измениться. Если spell не найден, будет fallback «не выбран», а found count окажется меньше total.

Пример E: изменить частоту обновления

Базовый tick задаёт kDiscordCallbackInterval. Например, одна секунда:

constexpr std::chrono::milliseconds kDiscordCallbackInterval{1000};

Но эта константа влияет сразу на три смысла: частоту RunCallbacks, частоту Presence polling и вычисление 10-секундного callback timeout/warning times. При изменении проверьте, что производные тики всё ещё дают желаемое время.

Более чистый способ разделить частоты — оставить Discord callbacks 500 мс и увеличить kPresencePollIntervalInCallbackTicks, например до 2. Тогда callbacks остаются быстрыми, а снимок читается раз в секунду.

Пример F: добавить новое поле в JSON

Допустим, нужен настраиваемый текст loading:

  1. Добавить std::string loadingText = "Загрузка"; в model::Config.
  2. Выбрать JSON-путь, например discord.loading_text.
  3. В ConfigLoader::Load вызвать существующий ReadString.
  4. В SendLoadingPresence использовать config_.loadingText.
  5. Добавить поле в config/DragonbornPresence.json.

Почему нужны все шаги: model задаёт default, loader валидирует внешний ввод, coordinator применяет значение, shipped JSON документирует рабочую настройку.

Пример G: добавить диагностическую запись 
SKSE::log::info(
    "Resolved display position: worldspace='{}' location='{}' cell='{}'.",
    worldspaceName,
    locationName,
    cellName);

Не ставьте такую запись в polling path на постоянной основе: она будет выполняться два раза в секунду даже без изменений. Для постоянного лога используйте существующий принцип «логировать только изменившийся Activity» либо уровень debug с отдельной настройкой logger.

Пример H: добавить новый Discord small asset state

Это сквозное изменение, поэтому одной строки недостаточно:

  1. Добавить default asset key в model::Config.
  2. Добавить поле в JSON и parser.
  3. Добавить состояние в PlayerSnapshot и прочитать его в StbDataProvider.
  4. Определить precedence рядом с combat в RefreshPresence.
  5. Передать выбранные smallImage/smallText в существующий payload.
  6. Загрузить Asset key в Discord Application.

Явно решите precedence. Например, combat должен перекрывать новое состояние или наоборот. Не размазывайте выбор по provider и client: provider сообщает факты, coordinator принимает display-решение, client только отправляет.

21. Инварианты: что нельзя случайно сломать

  • Исключения не выходят в Skyrim. Любой новый внешний callback получает собственную catch boundary.
  • Игровые данные читаются в game thread. Scheduler только ставит задачу.
  • В очереди не больше одной polling task. Не удалять coalescing без другой bounded-модели.
  • Только один UpdateActivity in flight. Pending signature блокирует второй запрос.
  • Timestamp стабилен весь запуск. Не вычислять current time внутри UpdateActivity или refresh.
  • Последний payload подтверждается callback. Не записывать last signature до Result::Ok.
  • UTF-8 режется только по границе code point. Не заменять на простой substr(0, 127).
  • Discord не устанавливается и не запускается. Loader только проверяет и безопасно отказывает.
  • После SDK failure нет повторных вызовов. Сначала health false, затем core reset.
  • Event sink возвращает kContinue. Другие плагины должны получить событие.
  • Config имеет defaults. Новое поле должно быть безопасно при отсутствии/ошибочном типе.
  • Более точные location rules идут раньше общих. Resolver использует first match.

22. Рекомендуемый маршрут изучения

  1. Откройте src/main.cpp рядом с разделом 8 и проследите только happy path.
  2. В DragonbornPresence.cpp прочитайте model; выпишите владельцев каждой строки и указателя.
  3. Проследите один RefreshPresence назад до ReadPlayerSnapshot и вперёд до Discord callback.
  4. Нарисуйте состояния loading_, gameLoaded_, active_.
  5. Пройдите scheduler: кто меняет каждый atomic и в каком потоке.
  6. Сломайте локальную копию одного необязательного JSON-поля и сопоставьте warning с loader-кодом.
  7. Добавьте одно тестовое location rule без C++, соберите и подтвердите large='...' в log.
  8. Только после этого меняйте формат Presence или источники STB.
