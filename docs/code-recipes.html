1. Безопасный цикл изменения

  1. Назовите observable result. Например: «в интерьере X Discord показывает asset Y».
  2. Найдите владельца решения. Game provider сообщает факты; coordinator выбирает отображение; client отправляет.
  3. Измените минимальное число слоёв. JSON-локация не требует C++.
  4. Соберите Release с PDB. Используйте текущий preset.
  5. Полностью перезапустите Skyrim. DLL и config не hot-reload.
  6. Пройдите изменённый путь. Не ограничивайтесь успешной компиляцией.
  7. Проверьте log и Discord. Snapshot доказывает queued payload; Presence updated — успешный callback.
Перед любым C++ изменением: сохраните текущий рабочий ZIP. Ошибка нативной DLL потенциально влияет на весь процесс Skyrim.

2. Добавить изображение локации без C++

config/DragonbornPresence.json

{
  "cell": "MySpecialInteriorCell",
  "image": "my_special_asset",
  "text": "Особое место"
}

Куда вставлять

Выше общего правила, которое также совпадает. Resolver прекращает поиск на первом match.

Почему только JSON

Model, parser и resolver уже поддерживают cell. Новый record — данные существующего алгоритма.

Проверка

  1. Asset key существует в Discord Application.
  2. Собрать, чтобы config скопировался в MO2, либо вручную изменить runtime JSON.
  3. Перезапустить Skyrim и войти в cell.
  4. Log: large='my_special_asset'.

3. Изменить формат state

src/DragonbornPresence.cpp PresenceCoordinator::RefreshPresence

const std::string stateText = std::format(
    "Ур. {} | Смерти: {} | {}",
    snapshot.level,
    deathsText,
    snapshot.stone);

Форматирование остаётся в coordinator, потому что это presentation policy. Provider не должен знать строку Discord, client не должен знать meaning level/deaths.

Конструкция

std::format проверяет число placeholders и types на уровне format API. Result владеет буфером до завершения UpdateActivity.

Проверка

Открыть save, увидеть точную строку, проверить кириллицу и очень длинное имя камня: client обязан обрезать по UTF-8 boundary.

4. Добавить настраиваемый loading text

model::ConfigConfigLoaderSendLoadingPresenceconfig JSON

Шаг 1: default

struct Config {
    // ...
    std::string loadingText = "Загрузка";
};

Шаг 2: JSON

"discord": {
  "enabled": true,
  "application_id": "...",
  "loading_text": "Пробуждение Драконорождённого"
}

Шаг 3: parser

ReadString(
    *discordConfig,
    "loading_text",
    config.loadingText,
    "discord");

Шаг 4: использование

config_.loadingText, // state
// ...
config_.loadingText  // small hover

Default обязателен: старый JSON должен продолжить работать. std::string нужен, потому что значение приходит из runtime JSON.

Проверка ошибок

Проверить отсутствующее поле, правильную string и неверный number. Последний должен дать warning и оставить default.

5. Добавить шестой режим сложности

model::DifficultyDifficultyName

enum class Difficulty : int {
    // 0–4...
    kSurvivalChallenge = 5,
};

case model::Difficulty::kSurvivalChallenge:
    return "Выживание";

Это корректно только если Papyrus/STB реально записывает 5. C++ enum сам не создаёт новую сложность в MCM.

Проверка

Выбрать режим через реальный STB UI. Snapshot log должен показать difficulty='Выживание', Discord details — то же значение. Значение 6 должно всё ещё давать не определена.

6. Добавить новый камень

kStoneDefinitions

StoneDefinition{
    "aaMy_NewDoomstoneDescription",
    "Новый камень"
},

Provider автоматически:

  1. увеличит std::array deduction size;
  2. найдёт spell на DataLoaded;
  3. попытается разобрать description.name;
  4. добавит spell в HasSpell search;
  5. использует fallback при проблеме description.

Порядок

Если одновременно активны два stone spells, выигрывает первый definition. Поставьте новый элемент в сознательную позицию.

Проверка

Init log denominator увеличился; numerator тоже увеличился, если Editor ID правильный. Затем получить spell и проверить state.

7. Изменить частоту snapshot без замедления callbacks

constants

constexpr std::uint8_t kPresencePollIntervalInCallbackTicks = 2;

При callback interval 500 мс snapshot будет раз в 1 секунду, но Discord RunCallbacks останется раз в 500 мс. Это лучше, чем менять базовый interval: timeout callback и warning delay сохраняют исходную временную семантику.

Tradeoff

Меньше game reads, но location/combat update может задержаться до секунды. Combat events всё равно только ставят flag и ждут poll.

Проверка

Сменить локацию и измерить по log/Discord. Убедиться, что UpdateActivity callback не получает ложный 10-секундный timeout.

8. Добавить small asset для скрытности

Это сквозной пример. Точные Skyrim API скрытности зависят от выбранного observable состояния; сначала подтвердите подходящий CommonLib getter для текущей версии.

Модель и config

struct Config {
    std::string stealthImage = "stealth";
};
struct PlayerSnapshot {
    bool sneaking = false;
};

Parser

Добавить assets.small_images.stealth через существующий ReadString.

Provider

В ReadPlayerSnapshot записать фактическое состояние игрока в sneaking. Provider сообщает факт, не Asset key.

Precedence в coordinator

std::string_view smallImage;
std::string_view smallText;
if (snapshot.inCombat) {
    smallImage = config_.combatImage;
    smallText = snapshot.combatText;
} else if (snapshot.sneaking) {
    smallImage = config_.stealthImage;
    smallText = "Скрывается";
}

Явный if/else if документирует решение: combat сильнее stealth. Не вычисляйте precedence внутри Discord client.

Внешняя зависимость

Asset stealth должен быть загружен в Discord Application; JSON не создаёт asset.

9. Добавить selector по plugin имени текущей cell

Пример показывает изменение алгоритма, а не только данных.

  1. Добавить std::string plugin в LocationImageRule.
  2. Прочитать optional string в ReadLocationImageRules.
  3. Учесть поле в hasSelector.
  4. В RuleMatches определить owning file текущей cell через подтверждённый CommonLib API.
  5. Сравнить имя точно и вернуть false при несовпадении.
  6. Документировать JSON schema.
{
  "plugin": "MyLocations.esp",
  "cell": "MyInterior",
  "image": "my_asset"
}
Почему здесь нет выдуманного getter: API владения form файлом нужно проверить в используемой CommonLibSSE-NG 3.7.0. Не угадывайте method name; compile success и runtime form inspection обязательны.

10. Добавить включаемый debug log

Не пишите info на каждом 500-мс poll. Добавьте config bool:

struct Config {
    bool verboseLogging = false;
};

Прочитайте через ReadBool. В интересующем блоке:

if (config_.verboseLogging) {
    SKSE::log::info(
        "Polling unchanged player state: level={} location='{}'.",
        snapshot.level,
        snapshot.location);
}

Лучше расширить logger level и использовать debug, но текущий logger настроен на info, поэтому debug сообщения иначе отбрасываются. Если меняете level, убедитесь, что release default остаётся info.

Проверка

false → файл не растёт на idle. true → ожидаемая строка примерно дважды в секунду. После диагностики вернуть false.

11. Сделать schema_version реальной проверкой

Сейчас loader игнорирует это поле. Для строгой версии:

constexpr int kSupportedSchemaVersion = 1;
const auto version = root.find("schema_version");
if (version == root.end() || !version->is_number_integer() ||
    version->get<int>() != kSupportedSchemaVersion) {
    SKSE::log::error("Unsupported config schema version.");
    return {};
}

Решение policy

Нужно заранее выбрать: отсутствующая версия считается legacy schema 1 или ошибкой. Строгий вариант сломает старые пользовательские JSON. Для внутреннего STB repository чистый cutover допустим, если shipped/runtime configs обновляются одновременно.

Проверка

Версия 1 работает; 2 и string "1" дают ожидаемый error и defaults.

12. Вынести форматирование Presence в чистую функцию

Если полей станет больше, можно отделить pure formatting от coordinator:

struct FormattedPresence {
    std::string details;
    std::string state;
};

[[nodiscard]] FormattedPresence FormatPresence(
    const model::PlayerSnapshot& snapshot)
{
    const std::string deaths = snapshot.deaths
        ? std::to_string(*snapshot.deaths)
        : std::string(constants::kUnknownDeathsText);
    return {
        snapshot.difficulty,
        std::format("lvl-{} 💀-{} {}",
                    snapshot.level, deaths, snapshot.stone)
    };
}

Это оправдано, если formatting растёт или получает несколько consumers. Для текущих двух строк существующий inline-код проще; не создавайте abstraction только ради количества файлов.

Преимущество

Pure function не обращается к Skyrim/Discord, поэтому её легче проверить обычными inputs. Но тест должен защищать observable contract, а не повторять implementation.

Дополнительный пример: показывать локацию в details

PresenceCoordinator::RefreshPresence

const std::string detailsText = snapshot.location.empty()
    ? snapshot.difficulty
    : snapshot.location;

Это минимальное presentation-изменение: provider уже отдаёт location, client уже ограничивает UTF-8 длину. Ternary сохраняет difficulty как fallback для пустой позиции.

Tradeoff

Discord имеет только ограниченный набор полей. После изменения выбранная сложность исчезнет из details; прежде чем переносить её в state, проверьте общий лимит и читаемость строки на телефоне.

Проверка

Проверить exterior, именованный interior и cell без display name. При переходе snapshot log должен показать новую location, затем Discord callback должен подтвердить update.

14. Чек-лист готового изменения

  • Новый field имеет безопасный default.
  • Неверный внешний input не заменяет default.
  • Ownership каждой строки/pointer понятен.
  • Skyrim calls остаются game-thread.
  • Новый callback имеет catch boundary.
  • Очередь работы остаётся bounded.
  • Timestamp не пересоздаётся.
  • Duplicate suppression учитывает все новые Discord fields.
  • Добавленный Asset key существует в Discord.
  • Shipped JSON и MO2 runtime JSON синхронизированы.
  • Release DLL и PDB собраны вместе.
  • Изменённый путь реально пройден в Skyrim.
  • Log содержит ожидаемый queued snapshot и успешный callback.
