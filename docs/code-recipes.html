Как расширять плагин без нарушения границ

Каждый рецепт начинается с observable result и проводит данные по существующему маршруту. Новый слой или abstraction нужен только при появлении новой внешней границы.

minimal pathowned valuescontract testsruntime smoke

1. Безопасный цикл изменения

  1. описать точный пользовательский результат и fallback;
  2. найти владельца решения: core, coordinator или adapter;
  3. изменить минимальный data path;
  4. проверить все приоритеты и transitions;
  5. добавить тест observable contract;
  6. собрать production target;
  7. проверить RE/SKSE/Discord path в STB, если он изменён;
  8. обновить shipped JSON и документацию.

2. Добавить новый параметр Presence

Если значение можно перечитать каждые 500 мс, отдельный Skyrim event не нужен.

include/DragonbornPresence/core/PlayerSnapshot.h
  ↓ новое owned поле
src/adapters/SkyrimTrueBeliever/StbGameDataSource.cpp
  ↓ чтение RE/STB и копирование
src/application/PresenceCoordinator.cpp
  ↓ формат, fallback и приоритет
tests/PresenceCoordinatorTests.cpp
  ↓ итоговый payload

Используйте std::optional<T>, если отсутствие отличается от нулевого значения. Не помещайте RE pointers в snapshot. Важное новое значение добавьте в диагностическую строку успешного update.

3. Добавить Skyrim event

Сначала решите, является ли событие длительным состоянием или только причиной перечитать snapshot.

СмыслApplication APIХранение
Диалог открыт/закрытOnDialogueChanged(bool)Состояние coordinator.
Повышение уровняRequestLevelRefresh()Pending flag до следующего tick.
Transient имя убитой целиOnTargetKilled(std::string)Owned application value.

Не передавайте RE::SomeEvent coordinator. Новый sink добавляется в StbRuntimeAdapter.h/.cpp, живёт полное время runtime, регистрируется через правильный event source и имеет exception boundary.

Если events только просят refresh, coalesce-ите их. При нескольких типах используйте bounded bit mask с явным приоритетом вместо отдельных SKSE tasks.

4. Добавить JSON-поле

core::Config safe default
  → JsonConfigProvider type-checked read
  → config/DragonbornPresence.json shipped value
  → JsonConfigProviderTests valid/missing/invalid cases
  → README и web documentation

Не увеличивайте schema_version при обратносуместимом optional поле. Повышение версии требуется только при несовместимой смене semantics или структуры. Provider сначала проверяет schema, а затем читает остальные разделы.

5. Добавить новое поле Discord SDK

Если меняется только текст существующих details/state/largeText/smallText, Discord adapter трогать не нужно. Для действительно нового SDK-поля:

  1. добавить engine-independent значение в PresencePayload;
  2. заполнить его в coordinator;
  3. сопоставить с discord::Activity;
  4. включить в activity signature;
  5. сохранить UTF-8 limits и one-in-flight guard;
  6. проверить duplicate suppression.
Частая ошибка: поле отображается при первом update, но последующие изменения подавляются, если его забыли включить в signature.

6. Добавить правило локации

Обычно меняется только config/DragonbornPresence.json:

{
  "worldspace": "DLC2SolstheimWorld",
  "location": "SomeLocation",
  "cell": "SomeCell",
  "match": "Отображаемое имя",
  "image": "discord_asset_key",
  "text": "Подсказка"
}

Нужен минимум один selector и непустой image. Все заданные selectors работают как AND. Более специфичное правило размещается раньше широкого. Exact duplicate selector делает более позднюю запись недостижимой.

После изменения JSON нужен перезапуск Skyrim, но не пересборка DLL. Asset key должен существовать в Discord Application.

7. Проверка результата

Автоматически

cmake --preset vs2026-tests
cmake --build --preset tests
ctest --preset tests
cmake --preset vs2026
cmake --build --preset release
powershell -File scripts/Validate-ReleaseArchive.ps1 \
  -ArchivePath build/vs2026/DragonbornPresence.zip

В игре

  1. загрузить плагин через SKSE;
  2. проверить регистрацию event sinks;
  3. вызвать изменённое состояние;
  4. сверить snapshot log и Discord Presence;
  5. проверить возврат в исходное состояние;
  6. убедиться, что одинаковый payload не отправляется повторно;
  7. проверить loading и Main Menu boundaries.
