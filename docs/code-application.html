Открыть реальный application layer

1. Forward declaration и event sink-классы

class PresenceCoordinator;

class MenuEventSink final
    : public RE::BSTEventSink<RE::MenuOpenCloseEvent> { ... };

class CombatEventSink final
    : public RE::BSTEventSink<RE::TESCombatEvent> { ... };

Forward declaration сообщает, что тип существует, до его полного определения. Этого достаточно для reference member PresenceCoordinator&, но не для хранения coordinator по value.

Public inheritance реализует интерфейс Skyrim observer. final запрещает дальнейшее наследование. Constructor принимает reference и записывает его member initializer list. Reference гарантирует обязательную связь, но sink не владеет coordinator.

ProcessEvent(... ) override заставляет compiler проверить совпадение с virtual signature базового класса. Второй параметр source не используется, поэтому в определении у него нет имени.

2. PresenceCoordinator как composition root

PresenceCoordinator() noexcept
    : menuEventSink_(*this), combatEventSink_(*this)
{}

Coordinator создаёт adapters и передаёт им reference на себя. *this — текущий объект. На стадии initializer list объект ещё строится, но reference только сохраняется; constructors sinks не вызывают методы coordinator.

Порядок фактической инициализации определяется порядком объявления fields, а не порядком в initializer list. В коде provider/client объявлены до sinks, затем sinks — это согласовано.

3. HandleException, HandleUnknownException и Stop

HandleException(context, details)

  1. active_=false закрывает новые work paths.
  2. permanentlyStopped_=true запрещает повторную регистрацию.
  3. callbackThread_.request_stop() ставит stop request.
  4. Critical log строится внутри собственного try.
  5. Discord client получает shutdown reason.

Параметр details nullable; ternary выбирает fallback. context — view, потому что callers передают literals.

HandleUnknownException

Не дублирует state transitions: делегирует общей функции со стабильным текстом.

Stop()

Private короткий метод используется после Discord failure. Он выключает coordinator и thread, но не повторяет transport log: client уже записал конкретную причину.

HandleCombatEvent()

  1. Игнорирует inactive, not loaded и loading.
  2. Проверяет, является ли actor или targetActor player.
  3. Отдельно обнаруживает возможное завершение боя: new state None и прошлый snapshot был combat.
  4. Ставит atomic combatRefreshRequested_=true.

Event не читает полный snapshot и не вызывает Discord. Несколько событий схлопываются в один bool.

SetLoading(bool)

Сначала записывает field. True отправляет loading Activity. False отправляет полный refresh только если gameLoaded. Это centralizes side effects смены loading.

7. StartCallbackThread(), строки 1359–1441

Однократный запуск

callbackThread_.joinable() true, если field уже связан с thread. Ранний return не создаёт второй scheduler.

Создание std::jthread

callbackThread_ = std::jthread(
    [this](std::stop_token stopToken) noexcept { ... });

jthread автоматически передаёт stop token callable, если сигнатура его принимает. Lambda захватывает coordinator pointer.

Цикл producer

  1. Пока stop не requested — sleep 500 мс.
  2. После sleep повторно проверяется stop.
  3. callbackTaskPending_.exchange(true) атомарно ставит флаг и возвращает прежнее значение.
  4. Если прежнее true, задача уже queued: увеличить wait ticks, иногда warning, continue.
  5. Если прежнее false, сбросить wait ticks и поставить ровно одну SKSE task.

Warning arithmetic

firstWarning = pendingTicks == 1;
repeatedWarning = pendingTicks > 1 &&
    (pendingTicks - 1) % 10 == 0;

Modulo выбирает ticks 11, 21, 31… после первого. Milliseconds = ticks × 500. Ternary добавляет plural suffix к английскому слову request.

Почему exchange, а не load/store

Два отдельных шага «прочитать false; записать true» не атомарны вместе. Exchange делает check-and-set одной atomic operation.

Queued main-thread lambda

taskInterface_->AddTask([this]() noexcept {
    callbackTaskPendingTicks_ = 0;
    callbackTaskPending_ = false;
    if (!active_) return;
    // RunCallbacks, poll, refresh
});

Флаг освобождается в начале исполнения, не в момент queueing. Затем:

  1. RunCallbacks; failure → Stop.
  2. Увеличить poll ticks.
  3. При threshold сбросить counter.
  4. Если loaded и не loading, atomic combatRefreshRequested_.exchange(false) выбирает reason combat/poll.
  5. Иначе повторно предложить loading Activity после завершения предыдущего pending callback.

Две exception boundaries

Внутренняя защищает main-thread task и вызывает полноценный HandleException. Внешняя защищает scheduler loop и вызывает HandleBackgroundException.

HandleBackgroundException

Ставит inactive, stop request, permanently stopped и critical log. Не вызывает discordClient_.Shutdown из background thread, чтобы destructor SDK core не исполнился там.

Основной thread invariant: scheduler не вызывает Skyrim getters и Discord core. Он только считает время и добавляет lambda через TaskInterface.

9. Реальные ProcessEvent, строки 1477–1505

Обе функции одинаково устроены: try, null check, forward; catch standard и unknown; всегда kContinue.

try {
    if (event) coordinator_.HandleMenuEvent(*event);
} catch (const std::exception& error) {
    coordinator_.HandleException("menu event handler", error.what());
} catch (...) {
    coordinator_.HandleUnknownException("menu event handler");
}
return RE::BSEventNotifyControl::kContinue;

*event превращает проверенный pointer в reference. kContinue не забирает событие у остальных registered sinks.

10. Глобальный объект и публичные обёртки

application::PresenceCoordinator g_presenceCoordinator;

Объект находится в анонимном namespace: один экземпляр на DLL, не exported. Его constructor выполняется при инициализации static storage, timestamp Discord client также создаётся один раз.

LoadConfig, RegisterGameEventHandlers, OnGameLoaded

Каждая public function:

  1. объявлена noexcept;
  2. вызывает один method глобального coordinator;
  3. ловит std::exception с конкретным context;
  4. ловит unknown;
  5. не возвращает exception в main.cpp/SKSE.

Это намеренное повторение на трёх ABI/event boundaries. Generic wrapper template сократил бы несколько строк, но усложнил бы beginner readability и context labels.
