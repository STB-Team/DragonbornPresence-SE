application · StbRuntimeAdapter

Координатор, события и граница потоков

PresenceCoordinator принимает решения без RE и Discord types. StbRuntimeAdapter переводит Skyrim events и планирует один bounded main-thread callback.

2. Запуск coordinator

Lifecycle разделён на этапы:

  1. LoadConfig() читает JSON до регистрации runtime;
  2. Start() всегда инициализирует game data source, а Presence transport — только когда интеграция включена и доступна;
  3. OnGameLoaded() отмечает доступное сохранение и публикует первый snapshot;
  4. Tick() обслуживает callbacks и периодический refresh;
  5. Stop() необратимо выключает текущий экземпляр.

Флаг permanentlyStopped_ не позволяет случайно перезапустить coordinator после transport failure.

3. Loading и gameplay states

loading_ и gameLoaded_ используются только на main thread. Правила:

  • до playable save публикуется стабильный loading payload;
  • открытие Loading Menu немедленно возвращает loading payload;
  • закрытие Loading Menu после загрузки вызывает gameplay refresh;
  • Main Menu сбрасывает признак загруженной игры;
  • gameplay snapshot не отправляется, пока loading active.

Application state не хранится в event sink: sink только сообщает изменение coordinator.

4. Формирование Presence

RefreshPresence(reason) выполняет один полный путь:

ReadPlayerSnapshot
  → deaths fallback
  → LocationAssetResolver
  → details/state strings
  → combat small asset
  → PresencePayload
  → IPresenceClient::UpdateActivity

Текущий контракт:

  • details и state независимо собираются из выбранных пользователем полей: сложности, уровня, смертей и камня;
  • large asset — первое matching location rule;
  • small asset — только loading или combat.

Подробный snapshot логируется только после принятого нового payload, поэтому неизменный polling не засоряет log.

5. Skyrim event sinks

StbRuntimeAdapter владеет двумя вложенными sinks:

Каждый ProcessEvent проверяет null, перехватывает все исключения и всегда возвращает kContinue. RE-event не пересекает application boundary.

6. Один scheduler, одна pending task

Background std::jthread не вызывает Skyrim forms или Discord SDK. Он только раз в 500 мс пытается поставить main-thread task через SKSE::TaskInterface.

scheduler thread                 Skyrim main thread
----------------                ------------------
callbackTaskPending == false
  → exchange(true)
  → AddTask ------------------→ clear pending
                                 PresenceCoordinator::Tick
                                 ReloadIfChanged
                                 RunCallbacks
                                 ReadPlayerSnapshot
                                 UpdateActivity
callbackTaskPending == true
  → увеличить wait counter
  → warning at 500 ms / 5 s

Combat events используют отдельный atomic request flag и объединяются ближайшим общим tick. Event storm не создаёт очередь из тысяч задач.

7. Остановка без разрушения Skyrim

Exception на внешней границе вызывает runtime handler:

  1. атомарно прекращает производство новых задач;
  2. просит coordinator остановить runtime только при внутренней C++-ошибке; transport failure ждёт явного retry;
  3. пишет context-specific critical log;
  4. оставляет Skyrim продолжать работу.

Background handler не уничтожает Discord core из scheduler thread. Окончательная очистка выполняется безопасным main-thread/application path и destructor order.
