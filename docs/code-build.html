CMake · CTest · GitHub Actions

Изолированные цели и проверяемый Release

CMake компилирует внутренние слои отдельно, собирает production SKSE-плагин и создаёт ZIP с точным manifest. CI проверяет и тестовые контракты, и настоящий runtime target.

2. Общие presets

Production configure:

cmake --preset vs2026

Release build:

cmake --build --preset release

Общий CMakePresets.json переносим: DRAGONBORNPRESENCE_DEPLOY_DIR по умолчанию пуст. Поэтому CI и новый clone только собирают и упаковывают проект, не записывая данные в локальный MO2.

3. Локальный deploy в MO2

Путь конкретного компьютера хранится в ignored CMakeUserPresets.json. Пример:

{
  "version": 6,
  "configurePresets": [{
    "name": "vs2026-local",
    "inherits": "vs2026",
    "cacheVariables": {
      "DRAGONBORNPRESENCE_DEPLOY_DIR": "D:/Mods/DragonbornPresence"
    }
  }],
  "buildPresets": [{
    "name": "release-local",
    "configurePreset": "vs2026-local",
    "configuration": "Release",
    "targets": ["DragonbornPresence"]
  }]
}

Использование:

cmake --preset vs2026-local
cmake --build --preset release-local

Post-build deploy копирует обе DLL, PDB и исходный JSON в SKSE/Plugins выбранного MO2 mod root.

5. CI для dev и pull requests

.github/workflows/ci.yml содержит два независимых jobs:

  1. Core, application и adapter tests: configure, build и CTest.
  2. Production SKSE plugin: production configure, Release build и строгая проверка созданного ZIP.

Второй job принципиален: unit-test targets не компилируют весь RE/SKSE/Discord runtime. CI теперь ловит ошибку outer adapter до создания релизного тега.

6. Release archive

Post-build команда создаёт build/vs2026/DragonbornPresence.zip. Скрипт scripts/Validate-ReleaseArchive.ps1 требует точный manifest:

SKSE/Plugins/DragonbornPresence.dll
SKSE/Plugins/DragonbornPresence.pdb
SKSE/Plugins/discord_game_sdk.dll
SKSE/Plugins/DragonbornPresence.json

Скрипт отклоняет отсутствующие, неожиданные, дублированные и пустые файлы, парсит JSON и выводит SHA-256 архива. Одна и та же проверка используется локально, в CI и перед GitHub Release.

7. Tag-gated GitHub Release

Workflow для тега v* выполняет полный gate:

  1. извлекает версию из project(DragonbornPresence VERSION ...);
  2. требует точное соответствие имени тега vX.Y.Z;
  3. собирает и запускает все тесты;
  4. собирает production Release;
  5. проверяет ZIP manifest и JSON;
  6. только затем вызывает gh release create.

Несовпадающая версия, failing test или неверный архив блокируют публикацию.
