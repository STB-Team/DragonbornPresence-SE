Что такое translation unit

После обработки #include препроцессором компилятор видит один большой текст — translation unit. Каждый .cpp компилируется отдельно в object file, затем linker соединяет object files и библиотеки в DLL.

#include <...> и #include "..."

Угловые скобки обычно ищут системные/настроенные include directories. Кавычки сначала ищут рядом с текущим файлом, затем в include paths. Это соглашение, а окончательный алгоритм задаёт компилятор.

extern "C"

C++ кодирует типы аргументов в имени символа — name mangling. C linkage просит использовать C-совместимое имя. Здесь функция не создаётся: объявляется символ __ImageBase, который предоставляет MSVC linker.

2. InitializeLogging(), строки 18–54

[[nodiscard]] bool InitializeLogging() noexcept
{
    try {
        const auto logPath = SKSE::log::log_directory().value_or(
            []() { /* fallback path */ }()) /
            "DragonbornPresence.log";

        auto sink = std::make_shared<spdlog::sinks::basic_file_sink_mt>(
            logPath.string(), true);
        auto log = std::make_shared<spdlog::logger>("global", std::move(sink));
        log->set_level(spdlog::level::info);
        log->flush_on(spdlog::level::info);
        spdlog::set_default_logger(std::move(log));
        return true;
    } catch (...) {
        /* debugger fallback */
        return false;
    }
}

Сигнатура

  • [[nodiscard]] предупреждает, если caller проигнорирует bool. Здесь результат определяет, можно ли продолжать загрузку.
  • bool: true — logger готов; false — запуск должен остановиться.
  • noexcept: функция обещает не выпустить исключение к SKSE loader. Обещание выполняется двумя catch-блоками.

Строки 21–33: выбор пути

SKSE::log::log_directory() возвращает optional path. value_or(fallback) берёт значение optional либо вычисленный запасной путь. Оператор / у std::filesystem::path добавляет компонент пути с корректным Windows separator.

Fallback реализован немедленно вызываемой lambda:

[]() {
    wchar_t buffer[MAX_PATH]{};
    const DWORD length = GetModuleFileNameW(
        reinterpret_cast<HMODULE>(&__ImageBase), buffer, MAX_PATH);
    if (length == 0 || length >= MAX_PATH) {
        return std::filesystem::current_path();
    }
    return std::filesystem::path(buffer).parent_path();
}()
ВыражениеПояснение
[]()Lambda без capture и аргументов.
Последние ()Lambda вызывается сразу; результат передаётся в value_or.
wchar_t buffer[MAX_PATH]{}Стековый массив wide chars; {} заполняет нулями.
&__ImageBaseАдрес внутри текущей DLL.
reinterpret_cast<HMODULE>Низкоуровневое переосмысление адреса как Windows module handle.

Тонкость: аргумент value_or вычисляется до вызова функции, даже если optional заполнен. Поэтому fallback lambda всё равно вызывается. Для старта плагина стоимость несущественна, но это полезно знать о eager evaluation.

Строки 35–41: создание logger

  • std::make_shared выделяет объект и control block shared ownership.
  • basic_file_sink_mt: mt означает thread-safe sink; сообщения могут идти из game и scheduler threads.
  • Второй аргумент true означает truncate? В spdlog для этого конструктора он является truncate; текущий код открывает файл с переданным значением. При чтении поведения ориентируйтесь на API версии 1.13.0.
  • std::move(sink) передаёт shared pointer в logger без лишнего увеличения/уменьшения ownership count в локальном объекте.
  • set_level(info) отбрасывает debug/trace.
  • flush_on(info) сбрасывает буфер для info и более серьёзных сообщений, чтобы данные пережили возможный crash.
  • set_default_logger делает logger доступным через logging facade.

Строки 42–53: два вида исключений

catch (const std::exception& error) сохраняет диагностический error.what(). catch (...) ловит всё остальное, включая исключения, не наследующиеся от std::exception. Logger ещё может быть нерабочим, поэтому используется OutputDebugStringA.

3. LogEntryPointFailure(), строки 56–67

void LogEntryPointFailure(const char* details) noexcept
{
    try {
        SKSE::log::critical(
            "DragonbornPresence entry-point failure: {} ...",
            details ? details : "unknown error.");
    } catch (...) {
        OutputDebugStringA(
            "DragonbornPresence: entry-point failure; plugin stopped.\n");
    }
}
БлокСмысл
Параметрconst char* — pointer на C-string; функция не меняет символы.
Ternarydetails ? details : ... не разыменовывает null pointer и выбирает fallback.
criticalОшибка произошла до регистрации нормальной игровой работы.
Внутренний catchДаже formatter/logger может бросить; второй канал диагностики не зависит от файла.

Функция не останавливает код сама: caller после неё возвращает false. Разделение позволяет всем ранним веткам иметь одинаковый текст ошибки.

4. OnSkseMessage(), строки 69–88

void OnSkseMessage(SKSE::MessagingInterface::Message* message) noexcept
{
    if (!message) {
        LogEntryPointFailure("SKSE delivered a null messaging event.");
        return;
    }

    using MI = SKSE::MessagingInterface;
    switch (message->type) {
    case MI::kDataLoaded:
        DragonbornPresence::RegisterGameEventHandlers();
        break;
    case MI::kNewGame:
    case MI::kPostLoadGame:
        DragonbornPresence::OnGameLoaded();
        break;
    default:
        break;
    }
}

Pointer и guard clause

SKSE передаёт pointer, поэтому теоретически значение может быть null. if (!message) — guard clause: аварийный случай обрабатывается в начале, основная логика остаётся без вложенности. message->type допустим только после этой проверки.

using MI = ...

Локальный type alias сокращает длинное имя. Он не создаёт новый тип и не выделяет память.

Объединённые case labels

kNewGame не содержит break и намеренно проваливается в kPostLoadGame. Оба события запускают одну логику. Это не случайный fallthrough: между labels нет statement.

Почему kDataLoaded отдельно

На стадии initial DLL load игровые data forms ещё могут быть не готовы. После kDataLoaded можно искать forms STB и регистрировать UI/combat sinks. Новая игра или loaded save сообщает, что уже нужен полный player snapshot.

default

Плагину не нужны остальные SKSE messages. Явный no-op делает выбор закрытым и не пишет лишний лог при каждом событии.

5. SKSEPluginLoad(), строки 92–146

SKSEPluginLoad(const SKSE::LoadInterface* skse)
{
    try {
        if (!skse) return false;
        SKSE::Init(skse);
        if (!InitializeLogging()) return false;
        // Discord availability
        // plugin declaration and version
        DragonbornPresence::LoadConfig();
        // messaging listener
        return true;
    } catch (...) {
        LogEntryPointFailure(...);
        return false;
    }
}

Что скрывает макрос SKSEPluginLoad

CommonLibSSE определяет корректную экспортируемую сигнатуру плагина. Исходно это выглядит как функция без вручную написанного return type, потому что макрос подставляет нужные declarations/attributes. SKSE находит экспорт по ожидаемому ABI.

Ранние возвраты

Каждый обязательный prerequisite использует return false сразу. В результате невозможно случайно зарегистрировать listener после неуспешного logger или считать plugin loaded без MessagingInterface.

Почему Discord проверяется ещё раз позже

Между SKSEPluginLoad и kDataLoaded проходит время. Discord может завершиться, поэтому DiscordPresenceClient::Initialize повторяет проверку непосредственно перед SDK creation.

Нельзя: бросать исключение из этой функции, возвращать true после частичной регистрации или читать PlayerCharacter здесь. Это слишком ранняя стадия Skyrim.

6. include/DragonbornPresence.h

#pragma once
#include <RE/Skyrim.h>
#include <SKSE/SKSE.h>

namespace DragonbornPresence {
void LoadConfig() noexcept;
void RegisterGameEventHandlers() noexcept;
void OnGameLoaded() noexcept;
}

Почему здесь только три функции

Header — внешний контракт translation unit. main.cpp не знает о PresenceCoordinator, StbDataProvider или Discord core. Это скрывает внутреннее устройство и сокращает compile dependencies.

Объявление и определение

Строка void LoadConfig() noexcept; объявляет сигнатуру и заканчивается ;. Тело находится в DragonbornPresence.cpp. Linker соединяет callsite с определением.

Namespace

DragonbornPresence::LoadConfig не конфликтует с возможной одноимённой функцией другого плагина или библиотеки. Namespace — compile-time организация имён, не runtime object.

#pragma once

Компилятор включает header один раз на translation unit. Это защищает объявления от многократной обработки через цепочку include.

7. Остальные маленькие заголовки

AdditionalFunctions.h

std::string Cp1251ToUtf8(const char* str);
bool IsValidUtf8(std::string_view str);

Первая функция возвращает владеющую строку, потому что создаёт новые байты. Вторая принимает невладеющий view, потому что только читает существующую последовательность.

discord_loader.h

namespace DragonbornPresence::detail {
[[nodiscard]] bool IsDiscordRunning(std::string* failureReason = nullptr) noexcept;
[[nodiscard]] bool LoadDiscordSdk(std::string* failureReason = nullptr) noexcept;
}
  • detail сообщает: это техническая часть, не основная бизнес-модель.
  • std::string* failureReason = nullptr — необязательный output parameter. Caller может получить подробность либо передать ничего.
  • [[nodiscard]] требует обратить внимание на успех/ошибку.
  • noexcept обязателен для ранней системной проверки.

pch.h

#pragma once
#include "SKSE/Impl/PCH.h"
using namespace std::literals;

Precompiled header заранее компилирует тяжёлые CommonLib declarations. using namespace std::literals делает доступными suffixes стандартных литералов. PCH ускоряет сборку, но не добавляет отдельный runtime-модуль.
