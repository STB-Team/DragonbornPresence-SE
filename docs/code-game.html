adapters/SkyrimTrueBeliever

Чтение Skyrim и данных сборки STB

StbGameDataSource — единственный источник gameplay snapshot. Здесь заканчиваются RE-указатели, Papyrus VM и строки игрового движка.

IGameDataSourceSTB.espPapyrus VMowned snapshot

1. Почему adapter называется SkyrimTrueBeliever

Интеграция опирается на конкретный STB.esp, его Form ID, Editor IDs, alias script и standing-stone description spells. Универсального skyrim adapter в проекте нет.

StbGameDataSource реализует application::ports::IGameDataSource:

virtual void Initialize() = 0;
virtual core::PlayerSnapshot ReadPlayerSnapshot() = 0;

Application видит только эти две операции и не включает RE/Skyrim.h.

2. Кэширование STB forms

Initialize() вызывается после kDataLoaded, когда TESDataHandler и plugin forms доступны. Метод один раз находит:

  • global смертей aaMZgv_NowDeath;
  • quest сложности из STB.esp;
  • description spell каждого поддерживаемого камня.

Указатели non-owning: формами владеет Skyrim. Отсутствующая форма остаётся null и позже даёт fallback. Adapter никогда не удаляет RE objects.

3. ReadPlayerSnapshot()

На каждом application poll создаётся новый owned snapshot:

PlayerCharacter
  → level
  → deaths global
  → selected stone
  → Papyrus difficulty
  → LocationContext
  → combat state and target text
  → core::PlayerSnapshot

Если player singleton отсутствует, возвращается default snapshot. Имя игрока намеренно не читается: оно не входит в текущий Presence-контракт и не должно создавать лишнюю строку каждые 500 мс.

4. Combat target

Когда игрок находится в бою, adapter пытается получить текущую цель и копирует её имя и уровень в lastCombatTarget_. Если Skyrim временно теряет target handle, последняя корректная цель сохраняет стабильную подсказку. После завершения боя cache очищается.

В бою с <имя цели> (ур. <уровень>)

Game data source сообщает факт inCombat и текст. Решение, показывать ли small combat asset, остаётся в coordinator.

5. Построение LocationContext

Adapter копирует:

  • Editor ID worldspace;
  • Editor ID текущей cell;
  • Editor IDs всей цепочки parent locations;
  • первое доступное локализованное имя location;
  • отображаемую комбинацию worldspace/location или fallback cell name.

Иерархия сохраняется в порядке обхода. Core resolver затем может сопоставить правило не только с текущей location, но и с её родителем.

6. PapyrusValueReader.h

Header-only helper изолирует механику Papyrus VM:

  1. получает VM policy;
  2. получает handle формы или alias;
  3. под lock находит attached scripts;
  4. сначала читает property, затем variable;
  5. преобразует поддерживаемый Papyrus type в C++ value;
  6. возвращает std::nullopt при отсутствии любого звена.

Шаблоны constrained допустимыми RE form/alias types. Координатор никогда не знает имени Papyrus script или property.

Открыть PapyrusValueReader.h

7. Кодировки строк STB

StbStringEncoding.cpp преобразует legacy CP1251 через два WinAPI-прохода:

CP1251 bytes → UTF-16 → UTF-8

FromGameString сначала принимает уже корректный UTF-8 без преобразования и только иначе вызывает Cp1251ToUtf8. Null, пустая строка или ошибка WinAPI дают пустую owned строку.

Граница: WinAPI-кодировка относится к STB adapter, а общая проверка и безопасное ограничение UTF-8 относятся к core.
